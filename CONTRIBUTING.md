# Contributing to Reproducibility Levels in Practice

Thank you for considering a contribution.

This repository is meant to be a living framework for discussing, refining, and extending practical levels of computational reproducibility. Contributions are welcome not only in the form of text edits, but also as critiques, examples, use cases, and suggestions for improving how the framework is used.

## Core contribution principles

This project works best when contributions are:

- constructive rather than punitive;
- practical rather than performative;
- accessible to researchers with different technical backgrounds;
- precise enough to be useful, while still readable and inviting.

## Before you contribute

Please take a quick look at:

- [`README.md`](README.md) for the overall project framing;
- the files in [`levels/`](levels/) for the level descriptions;
- [`GOVERNANCE.md`](GOVERNANCE.md) for maintainership, decisions, and versioning policy;
- [`reprolevel_manuscript.md`](reprolevel_manuscript.md) for the current manuscript draft.

Before proposing a major change, it is helpful to be explicit about whether the suggestion is primarily about:

- clarity;
- correctness;
- scope;
- tone;
- usability;
- disciplinary applicability.

That context makes discussion much easier.

## What kinds of contributions are helpful

Contributions that are especially useful include:

- clarifying the wording of an existing level;
- pointing out conceptual inconsistencies across levels;
- suggesting examples from specific disciplines or research settings;
- proposing improvements to self-assessment or teaching use;
- identifying where the framework is too strict, too vague, or too tool-centric;
- improving the manuscript framing or repository structure;
- fixing typos, formatting issues, broken links, or metadata errors.

## Contribution workflow

### For substantial changes

Substantial changes include:

- changes to level definitions;
- changes to the core framework concepts;
- changes to governance or versioning policy;
- additions of new assessment dimensions;
- major restructuring of repository content.

For these changes, please use this process:

1. Open an issue first to explain the proposal.
2. Discuss the proposal with maintainers and contributors.
3. Once there is enough clarity, open a pull request linked to the issue.
4. Address review comments and update the pull request as needed.
5. Wait for maintainer approval before merge.

Major changes require maintainer review and approval before they can be merged.

### For smaller changes

For small fixes such as typos, broken links, formatting improvements, or minor wording clarifications, you may open a pull request directly. If in doubt, open an issue first.

## Pull request expectations

Please aim to keep pull requests focused. Small, clear changes are much easier to review and discuss than very broad rewrites.

Each pull request should:

- explain what changed;
- explain why the change is useful;
- link to the relevant issue if one exists;
- update related documentation where needed.

If your pull request addresses a conceptual issue, please explain not only what changed, but also what problem the change is meant to solve.

## Writing style guide

To keep the repository coherent, please follow these writing conventions:

- Use **British English** spelling.
- Keep the tone constructive, respectful, and non-shaming.
- Prefer plain, direct language over jargon.
- Describe capabilities and outcomes, not just preferred tools.
- Keep examples realistic and grounded in research practice.
- Preserve the narrative accessibility of the framework while improving precision where needed.

Examples:

- prefer `organisation` over `organization`;
- prefer British spellings such as `behaviour`;
- prefer `licence` for the noun in prose where appropriate, while keeping formal licence names unchanged.

## Repository conventions

When editing level files, try to maintain the shared structure of:

- title;
- summary or framing;
- characteristics and behaviours;
- tools and artefacts;
- pros;
- cons or risks;
- how to recognise the level;
- how to move beyond it.

When updating governance, citation, or release information, please also check whether the same information appears in:

- [`README.md`](README.md);
- [`GOVERNANCE.md`](GOVERNANCE.md);
- [`CHANGELOG.md`](CHANGELOG.md);
- [`CITATION.cff`](CITATION.cff).

## Review and approval

Maintainers review pull requests for:

- conceptual fit;
- clarity and tone;
- internal consistency across the framework;
- style-guide compliance;
- documentation completeness.

At least one maintainer approval is expected for substantial changes.

## Questions

If you are unsure whether an idea fits, open an issue and describe it briefly first.

Thank you for helping make this framework clearer, fairer, and more useful.
