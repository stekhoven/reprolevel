# Level 2 – Carpe Diem

*Seize the day – write it down while it still works.*

## What does Level 2 look like?

At **Level 2**, you’ve begun to move beyond the purely ephemeral. This level is all about **seizing the moment of reproducibility**: you capture your analysis steps while they’re fresh, making it possible (in theory) to get the same results later *as long as you don’t wait too long*. The work is more orderly now – perhaps you save scripts or jot down methods as you go – but it’s still a far cry from fully automated reproducibility. Think of it as writing a rough recipe right after cooking a dish. You have something to refer back to, and if you try the recipe again tomorrow, you’ll probably succeed. But if you hand that scribbled recipe to a stranger, or even to yourself next year, the outcome might not be as consistent.

In practical terms, **Level 2 means your analysis now has some documentation and scripting, but with a short shelf-life.** You can likely re-run everything next week and get the same result (especially if nothing in your computing environment has changed). But as time passes or setups change, cracks will appear – missing dependencies, forgotten setup steps, or ambiguous instructions.

**Carpe Diem** (Latin for “seize the day”) is a fitting name: at this stage you’re capturing today’s work so it isn’t lost, but the approach is still very “live in the now.” Reproducibility is achievable, but often only in the short term or with your direct involvement.

## Typical characteristics and behaviours

*   **Some scripting or notebooks:** Unlike Level 1, you’re now writing code for your analyses, not just clicking through interfaces. You might have one or several scripts that perform key parts of the analysis (e.g. a script to clean data, another to make figures) or a Jupyter/R Markdown notebook blending code and narrative. These scripts/notebooks cover much of what you do – though a few manual steps might still lurk between them.
*   **Basic documentation:** You maintain at least a minimal record of what you’re doing. This could be a **README file** with instructions on how to run the analysis, comments in your code, or a simple lab notebook entry for the experiment. The documentation might be a bit terse or incomplete, but it exists and gives you something to consult later. (It may lag behind the code if you’re in a rush, but at least there’s some record beyond your memory now!)
*   **Ad-hoc organisation:** Your project folders have a clearer structure. You might separate raw data, scripts, and results into subfolders instead of everything being in one heap. Naming conventions are more thoughtful (files like `data_cleaned_v1.csv` or `analysis_final.Rmd` are common). It’s not a perfect system, but it’s far more organised than Level 1’s chaos.
*   **Environment mostly implicit:** You typically run your analysis on the same machine or environment where you developed it, and it works for you there. If you need a package or library, you install it on the fly (often without recording the version or exact install steps). You might keep a list of required software somewhere in your notes (“uses Python 3 and these R packages…”), but there’s no guarantee someone else could set up the environment the same way without your help.
*   **Early version control (maybe):** By this stage, you might have *dabbled* in version control. Perhaps you’ve put your code on GitHub or started using Git locally to track changes. However, this usage is basic – you commit infrequently or mainly use GitHub as a backup rather than an active collaboration tool. It’s not uncommon for Level 2 projects to still rely on manual versioning (e.g. dated file names or occasional zip backups) instead of a comprehensive Git history.
*   **Some collaboration, low bus-factor:** You may have begun sharing your work with colleagues. You provide them with your scripts, data, and a short how-to guide (maybe via email or a shared folder). They *might* get it working on a similar setup, but usually your guidance is needed to fully reproduce the results. The project’s “bus factor” is still low – if you disappeared, others would struggle, though not as disastrously as at Level 1.

## Typical tools and artefacts

*   **Script files and notebooks:** You now have `.R` scripts, `.py` scripts, or notebook files (`.ipynb`, `.Rmd`) as part of your project. These encode much of your analysis. For example, you might run an entire analysis by executing a Jupyter Notebook top-to-bottom (assuming it runs cleanly) or by running a master script that calls others in sequence.
*   **Project README or notes:** There’s likely a `README.md` or other notes file in your project that explains what the project is and how to run it, what data files are needed, and what output to expect. This provides a starting point for anyone (including you) to understand the workflow.
*   **Local knowledge base:** You might use an Electronic Lab Notebook (ELN) or a personal wiki to jot down important details (like dataset sources, filtering criteria, or analysis decisions). Or you might simply keep a Word document or OneNote page with experiment notes. The key point is you’re writing a lot more down than at Level 1, creating a knowledge trail.
*   **Version control snapshots:** If you’re using Git, you’ll have a repository (even if private) with a history of commits – though the commit messages might be uninformative, like “latest changes”. If you haven’t fully adopted Git, you probably have manual version snapshots: dated zip archives or files named `analysis_v1.R`, `analysis_v2.R`, etc. Either way, you’re no longer overwriting the only copy of important files; you have some record of past versions.
*   **Shared folders and cloud storage:** Collaboration or backup might be handled by a cloud folder (Dropbox, Google Drive, OneDrive, etc.). Your project could live in a synced drive, letting you access it from multiple devices or share it easily. This works, but since the structure isn’t enforced by tooling, it relies on trust and caution (one misplaced or deleted file by a collaborator could cause confusion).

## What works well at this level

*   **Short-term reproducibility:** Your analyses are **repeatable in the short term**. If you need to re-run an analysis next week or next month (on the same machine), you can usually do so with minimal fuss. This is a big improvement over Level 1 – you no longer feel like everything would fall apart if asked to show your work again. There’s relief in knowing you have code and notes saved to back you up.
*   **Improved reliability:** Because you’ve written down steps and created scripts, you’re less likely to make the exact same mistake twice. If a result looks odd, you can tweak the code and run it again to verify, rather than trying to retrace a complex manual process. Debugging and refining analyses are easier than at Level 1, since much of the process is captured in code.
*   **Faster updates:** Need to adjust a parameter or add new data? At Level 2, you can do that more efficiently. You change the code or notes where needed and re-run the analysis, rather than redoing everything from scratch. This means you can iterate on your research more quickly and respond to feedback (or reviewer comments) without starting over.
*   **Basic collaboration possible:** You can now share your work with others in a semi-usable form. Handing someone a script or notebook plus the necessary data is much better than giving only final results. If they’re somewhat tech-savvy (and their setup is similar to yours), they might get it running. Your work is no longer completely personal; others can, in principle, engage with it or build upon it, even if it’s a bit of a manual process.
*   **“Good enough” for many cases:** For many small or internal projects, Level 2 feels “good enough.” It meets basic reproducibility needs, and the overhead remains low – you haven’t had to invest in complex infrastructure, just a bit of effort to save code and notes. Often, that suffices to satisfy immediate requirements (like keeping a lab record or providing supplemental code for a paper).

## What can go wrong (risks and limitations)

*   **Reproducibility fades over time:** Over months or years, updates and environment changes can break what once worked. Because Level 2 doesn’t lock down environments or data versions, a script that ran fine last month might not run next year without troubleshooting. In short, the longer you wait (or the more things change), the shakier reproducibility gets.
*   **Incomplete documentation:** While you do have documentation now, it might not capture everything. Perhaps you updated the code but forgot to update the README instructions, or you assumed some steps were “obvious” and didn’t write them down. Others (or future you) won’t find those assumptions obvious at all. This can lead to confusion or errors, and extra back-and-forth to fill in the gaps.
*   **Manual steps still lurking:** It’s common at Level 2 that *most* of the process is scripted, but a few steps remain manual. For example, you might still manually download a dataset from a web portal, or click a button in a software tool to export a graphic. Those steps can be easy to overlook. If someone tries to run just your scripts, they might hit a wall because “Step 3: remove outliers in Excel” wasn’t documented or automated. In such cases, the reproducibility chain breaks at the weakest link.
*   **Version confusion:** If you’re not fully comfortable with versioning, you might end up with multiple copies of files and not be entirely sure which is the latest (e.g. `analysis_v2.R` vs `analysis_latest.R`). This can cause you to accidentally use outdated code or share the wrong file with colleagues. Small inconsistencies creep in when different team members – or your own devices – have slightly different versions of the project.
*   **False sense of security:** Because you have code and notes, it’s easy to assume everything is under control. But until there’s an independent re-run (by someone else or by you after some time), hidden reproducibility gaps could remain. In short, your documentation might not be as clear to others as it is to you, and the process might not be as foolproof as it feels. It’s only when someone truly “outside your head” follows the steps that you find out if anything was misunderstood or missing.

## How to know if you’re at Level 2

You might recognise **Level 2 – Carpe Diem** in your work if:

*   **You have scripts/notebooks for your analysis:** You’re no longer doing everything manually; at least some substantial part of the workflow is in code that you can re-run. If asked how you got a result, you could point to a script or notebook (even if you might need to explain a few caveats or extra steps).
*   **There’s a README or notes file in your project:** Your project folders aren’t just random files; there’s at least one document that describes what’s going on (even if only briefly) – for example, a README with setup instructions or a notebook that combines narrative with code.
*   **Some order in the chaos:** Your files are more organised now. You might have standard subfolders (data, scripts, outputs) or a naming scheme for files, instead of everything being in one place. It’s not perfectly tidy, but when you return to the project after a break, you can generally find things and understand what you did.
*   **Basic use of collaborative tools:** You’ve shared code with others – maybe on GitHub/GitLab or simply by emailing a zip file. Perhaps a colleague even managed to run your code with some guidance from you. It wasn’t a one-click process for them, but it did run outside of your own laptop, which is progress.
*   **No heavy automation or environment control (yet):** You aren’t using tools like containers, workflow managers, or automated tests (CI), and some steps are still manual. In other words, the advanced practices of Level 3 are still ahead of you.

If this sounds like you, then congratulations – you’re at Level 2. **That’s a good place to be.** It means you care about reproducibility and have already improved on the completely ad-hoc approach. Many researchers operate comfortably at this level, especially for smaller or short-term projects. But if you’re looking to further bulletproof and share your work, read on for what Level 3 has in store.

## Moving beyond Level 2

When you’re ready to advance to **Level 3 (Sweet Spot)**, the goal is to make your work more robust and less dependent on time and context. Here are some gentle steps to start moving in that direction:

*   **Formalise version control:** If you haven’t fully embraced Git yet, now’s the time. Start using Git in earnest: commit changes with descriptive messages, push your work to a remote repository (e.g. GitHub) for safekeeping, and learn to use branches for new features or experiments. A reliable version history of your project will reduce confusion over which version of a file is the “right” one and when changes were introduced.
*   **Automate the easy parts:** Look at your workflow and identify any remaining manual steps – then script whichever you can. For example, if you manually download data, write a small script (or use an API) to fetch it; if you copy results into a spreadsheet for final tweaks, see if your code can produce the polished output directly. You don’t have to automate everything at once – tackle the low-hanging fruit first, removing a few manual tasks step by step.
*   **Capture your environment (lightly):** Begin recording details about your software environment. In R, you might use `renv` or at least note package versions in your README; in Python, you could create a `requirements.txt` or export a Conda `environment.yml`. Even a basic list of packages and versions can save headaches later. This is a stepping stone towards the more rigorous environment management at Levels 3 and 4.
*   **Improve documentation continuously:** Treat documentation as a living part of the project, not a one-time task. Make it a habit to update your README or notebook text whenever the analysis changes. Imagine a future reader (or yourself in six months) trying to follow along – ensure the instructions and descriptions are up to date and clear for that person.
*   **Dry-run on a clean setup:** Test your analysis on a clean environment to catch hidden dependencies. For instance, try running the whole pipeline on a different computer or a fresh virtual environment where you haven’t pre-installed everything. This is like a manual “integration test” for your project. Note what breaks or what’s missing, then adjust your code or documentation to fix those issues.
*   **Consider the “bus factor”:** Think about handing off your project. Write a short guide on how someone else could continue this work or reproduce it without you (e.g. “How to add new data and rerun the analysis” or “Steps to extend this for next year’s study”). This not only helps others, it often clarifies the process for you. It’s an intermediate step towards the fully shareable, on-board-anyone workflows that define Level 3 and beyond.

By gradually adopting these practices, you’ll find your project becoming more **repeatable, shareable, and resilient**. Level 3 will introduce even more powerful techniques (like fully embracing workflow automation, collaborative coding practices, and more thorough environment management), but you don’t need to leap there overnight. Each incremental improvement you make at Level 2 – whether it’s using Git more effectively, scripting one more task, or listing your software dependencies – builds momentum towards highly reproducible research. *Carpe diem*, and onward to Level 3 when you’re ready!
